# Focus Bracket Fixture Manifest

Date: 2026-06-14
Scope: GitHub issue #192, focus bracket fixture set

## Purpose

Focus stacking validation needs fixture coverage before runtime output can be
called reliable. This manifest defines the fixture categories, metadata, review
uses, and asset policy for future focus stack tests without adding large image
assets to the repository yet.

## Fixture Categories

### Synthetic Tiny Bracket

Purpose: deterministic CI smoke once runtime exists.

Required coverage:

- three focus slices;
- simple foreground, midground, and background targets;
- known output dimensions;
- stable synthetic source hashes;
- no licensing ambiguity.

Validation uses:

- dry-run schema roundtrip;
- apply smoke for accepted dry-run plan;
- source index ordering;
- memory/runtime Tier 1 budget checks;
- stale-source apply rejection.

### Macro Product Bracket

Purpose: product/photo workflow representative validation.

Required coverage:

- fine printed text or label edges;
- high-frequency product texture;
- specular highlight transitions;
- shallow depth of field;
- 5 to 12 focus slices.

Validation uses:

- sharpness map review;
- focus transition review;
- rejected-frame review;
- retouch-layer artifact review;
- output crop comparison.

### Natural Texture Bracket

Purpose: detect texture collapse and false sharpness.

Required coverage:

- foliage, bark, fabric, hair, or similar natural detail;
- depth transitions across multiple planes;
- low-contrast texture;
- optional high-ISO variant.

Validation uses:

- texture preservation review;
- halo/ringing review;
- local blend artifact review;
- performance Tier 2 evidence.

### Parallax And Motion Stress Bracket

Purpose: block unsafe or misleading focus stack applies.

Required coverage:

- foreground/background parallax;
- slight camera movement or handheld drift;
- moving subject or unstable detail;
- hard subject boundaries.

Validation uses:

- alignment warning stability;
- rejected-frame reasons;
- apply blocking/downgrade decisions;
- manual review checklist coverage.

## Required Fixture Metadata

Every fixture set must include:

- fixture ID and version;
- source file IDs and stable content hashes;
- source graph revisions or fixture graph revision;
- capture device/lens metadata when known;
- source dimensions and color space;
- focus slice order and optional focus distance;
- license and provenance statement;
- allowed validation uses;
- disallowed validation uses;
- expected warning/block codes;
- expected artifact types: sharpness map, depth/confidence map, preview, final
  output, and optional retouch layer.

## Asset Policy

- Do not commit large RAW/photo fixtures directly until the fixture download,
  license, hash, cache, and privacy policy covers them.
- Synthetic tiny fixtures may be generated by script once the runtime exists.
- Real-photo fixtures should live in a documented cache/download path with
  content hashes and license allowlist.
- Fixtures must not include private client work, faces without explicit consent,
  or unknown-license internet images.

## RAW Corpus Placeholders

Issue #1899 reserves focus-stack RAW corpus slots:

- `real.focus.macro-bracket-flower.v0`
- `real.focus.product-depth-stack.v0`
- `raw-evidence.focus-stack.product-depth.v1`

These entries are schema/metadata only. They do not add RAW sequences, focus
maps, blend outputs, source hashes, or quality claims. A later PR must attach
approved rights, hashes, ordered-source metadata, render artifacts, and review
evidence before these entries can count as runtime focus-stack proof.

## Current Status

The first deterministic synthetic fixture manifest and generator live at:

- `fixtures/focus-stacking/focus-synthetic-bracket-fixtures.json`
- `tests/integration/checks/check-focus-synthetic-fixtures.ts`

They generate tiny temporary PPM sources during validation and verify stable
fixture hashes. They do not commit raster assets or claim runtime focus-stack
quality. Runtime focus stack tests should consume these fixtures after the
plan-only preflight, alignment, sharpness-map, and blend paths exist.
